## 项目背景

燃料电池作为清洁能源的重要载体，已广泛应用于新能源汽车、分布式发电等领域。其核心组件——膜电极（MEA）、双极板（BPP）等的装配质量直接影响电池性能、寿命和安全性。

传统的人工目检方式存在三大痛点：

- 👁 **效率低下**：单个组件的检查需耗时数分钟
- 🎯 **一致性差**：不同检测人员判断标准不一
- 😴 **疲劳衰减**：长时间工作后漏检率显著上升

本系统基于深度学习技术，实现了一套全自动的燃料电池组件缺陷检测方案。

## 缺陷类型与检测策略

本系统基于 YOLO 实例分割模型，能同时完成**组件识别**和**缺陷分类**：

### 四种缺陷类型

| 缺陷类型 | 描述 | 检测策略 |
|----------|------|----------|
| 🔴 **漏装** | 指定位置缺少组件 | 目标位置无检测框 → 标记漏装 |
| 🟡 **多装** | 多余组件出现 | 检测框数量超过预期 → 标记多装 |
| 🟠 **错装** | 组件位置不正确 | 检测框类型与预期位置不匹配 → 标记错装 |
| 🔵 **倒装** | 组件方向错误 | 分割 mask 角度分析 → 标记倒装 |

### 检测流程

```
图像输入
  │
  ▼
预处理（缩放、归一化、数据增强）
  │
  ▼
YOLO 实例分割推理
  │
  ├─→ 组件实例分割 → 获取每个组件的 Mask + 类别 + 位置
  │
  ▼
组件排列顺序分析
  │
  ├─→ 与标准模板比对 → 判定装配序列是否正确
  │
  ▼
缺陷判定逻辑
  │
  ├─→ 漏装 / 多装 / 错装 / 倒装 分类
  │
  ▼
生成检测报告 → 存入数据库 → 返回前端可视化展示
```

## 技术选型说明

### 前端：React 18 + TypeScript

选择 React 18 的原因：
- **类型安全**：TypeScript 在大型项目中减少类型错误
- **状态管理**：Zustand 轻量级方案，比 Redux 更简洁
- **动画交互**：Framer Motion 实现检测过程流畅动画
- **样式方案**：Tailwind CSS 实现快速响应式布局，无需手写大量 CSS

### 后端：FastAPI (Python)

选择 FastAPI 的原因：
- **高性能**：基于 Starlette 异步框架，并发能力出色
- **自动文档**：内置 Swagger UI，接口调试方便
- **类型校验**：Pydantic 模型自动验证请求数据
- **与 Python ML 生态无缝衔接**：直接调用 PyTorch/ONNX 模型

### 模型选型

- **基础模型**：YOLOv8-seg（实例分割版本）
- **选择原因**：相比 Mask R-CNN，YOLOv8 在速度和精度之间取得更好平衡
- **数据集**：自制数据集，包含 5000+ 标注样本，涵盖正常装配和各类缺陷
- **数据增强**：随机翻转、旋转、亮度调整、Mosaic 拼接
- **训练策略**：迁移学习（COCO 预训练权重）+ 两阶段微调

### 训练流程

| 阶段 | 训练内容 | 轮数 | 学习率 |
|------|---------|------|--------|
| 阶段一 | 冻结 Backbone，仅训练检测头 | 50 epoch | 0.01 |
| 阶段二 | 全网络微调 | 100 epoch | 0.001 |
| 阶段三 | 针对性数据增强训练 | 50 epoch | 0.0005 |

## 数据库设计要点

系统的数据表设计围绕三个核心实体展开：

- **检测记录表**：存储每次检测的原始图像、检测时间、处理耗时
- **缺陷明细表**：关联检测记录，记录每个缺陷的类型、位置、置信度
- **模型版本表**：跟踪模型迭代，记录模型文件路径、训练时间、评估指标

数据库选用 MySQL，主要考虑：
- 结构化数据存储需求明确
- 事务支持保证数据一致性
- 丰富的索引机制加速查询（按时间、缺陷类型等维度）

## 技术挑战与解决方案

| 挑战 | 解决方案 |
|------|---------|
| 小目标检测精度低 | 增加高分辨率输入 + FPN 多尺度特征融合 |
| 数据标注成本高 | 半自动标注工具 + 数据增强扩充样本 |
| 推理速度要求高 | ONNX 模型导出 + TensorRT 加速 |
| 不同类型缺陷样本不均衡 | Focal Loss + 过采样少数类 |
| 前后端联调效率低 | FastAPI 自动生成 Swagger 文档 |
